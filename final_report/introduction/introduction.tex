\chapter{Introduction}
\section{Background}


Refactoring is a routine part of software development. Developers use it to improve the structure of code (for example its readability, maintainability, or ability to evolve) without changing the behaviour that users or other systems observe. Classic definitions describe refactoring as a sequence of small, semantics-preserving transformations \cite{opdyke1992refactoring,fowler1999refactoring,fowler2018refactoring}, and later work organises the main refactoring operations and the tools that support them \cite{mens2004survey}. In practice, refactoring is encouraged because poorly structured code makes later enhancement and maintenance harder, and long-lived systems need continual adjustment as they change. It is not cost-free, however: it takes developer time, can introduce risk, and is often mixed into feature work or bug fixing rather than carried out as a separate ``refactoring-only'' task \cite{murphyhill2009howrefactor,kim2011apilevel}.

\section{Motivation}

Most research and tooling that evaluates refactoring focuses on \textit{outcomes} rather than the \textit{process}. The usual approach is to compare a ``before'' and ``after'' version of the code using static quality indicators such as code smells, cohesion, coupling, complexity, or other maintainability-related metrics. This appears in work that detects or prioritises candidate refactorings, and in recommendation systems that rank possible changes by their expected effect on design quality. Other work broadens the evidence used for ranking, for example by adding traceability information, or by studying how repository-level product and process metrics relate to developers' reasons for refactoring. In parallel, refactoring-detection tools have made it easier to recover refactoring operations from code changes, especially through AST differencing and structured matching. Tools such as RefactoringMiner have therefore enabled larger empirical studies of when refactoring happens, which kinds are used, and what development contexts they appear in \cite{alikhanifard2025rminer,falleri2024gumtree}.

These perspectives all evaluate refactoring as a mapping from an initial program state to a final program state, leaving the \textit{trajectory} between those states largely outside the picture. Developers rarely refactor in a single atomic operation. A rename may be half-applied before the tests are run, or an extracted method may introduce a temporary structure that is cleaned up only later. Two sessions can therefore end in the same code while placing very different costs on the developer along the way. Existing endpoint-only evaluation cannot distinguish these cases. Research on refactoring processes also suggests that refactoring is often ``flossed'' into ongoing development rather than carried out as a separate dedicated task, which increases the chance of mixed-intent steps and noisy traces \cite{murphyhill2009howrefactor}. The quality of the process, that is, how a developer navigates the intermediate states, therefore matters for both developer productivity and the health of the codebase as it evolves.

Some recent work does look at refactoring trajectories more directly. RefactorBench uses a state/action/observation framing to study failures by language-model agents on multi-file refactoring tasks \cite{gautam2025refactorbench}, and Bouzenia et al.\ study recurring action patterns in software-engineering agent traces \cite{bouzenia2025trajectories}. A recent review of LLM-based refactoring work also discusses evaluation methods and open problems in the area \cite{alomar2025slr}. These works are close to the problem studied here, but they still do not directly evaluate the path taken in an observed developer session. The nearest systems either generate a path to a desired end state, without judging the developer's own path, or recommend refactorings towards an architectural target, without starting from an observed trace. This project focuses on that missing case: it scores recorded refactoring traces, constructs higher-scoring reference trajectories, and identifies the points where the recorded trace diverges from them.

\section{Research Objectives}

This thesis asks whether, given the start and end state of a refactoring session, we can evaluate the developer's steps between them and identify points where an alternative route would have scored better. The core idea is to model refactoring as a \textit{process} over states, rather than judging it only by comparing endpoints. To do this, we use a state/action/trace framing: snapshots of the codebase are treated as \textit{states}, and developer edits (including IDE refactorings and manual transformations) are treated as \textit{actions} that produce a trace from the start state to the end state.

The first component is a \textbf{process-quality metric} that scores traces according to criteria drawn from existing research on code-quality metrics, smell-detection validity, and process-level signals. The metric is not purely structural: it combines several terms covering endpoint improvement, intermediate-state degradation, churn and disruption, and safety-related signals such as preserving build and test behaviour. Combining several signals is deliberate, since any single signal can be misleading on its own.

Scoring the observed trace is only part of the approach. We also construct \emph{counterfactual} alternatives: \textbf{reference trajectories} between the same start and end states that score better under the chosen process-quality metric. The observed trace is then compared with these alternatives to find \emph{divergence points}, where the developer's route moves away from a higher-scoring one, and to estimate how much each divergence contributes to the final quality gap. This connects to systems that generate or recommend refactoring sequences. ReSynth synthesises a path to a target end state from a developer's partial edits \cite{raychev2013resynth}, Refactoring Navigator recommends steps towards an architectural target \cite{lin2016navigator}, and search-based refactoring treats sequences as an optimisation problem under explicit quality functions \cite{harman2007pareto,mohan2019manyobjective}. In those systems, the generated sequence is itself the output. In this project, it is used as a comparison point for evaluating the process the developer actually followed.

\section{Contributions}

The main contributions of the project are:

\begin{itemize}
  \item A process-quality metric for refactoring trajectories. It combines endpoint cleanliness gain with process terms such as intermediate-cleanliness lag, so two sessions that finish in the same code can still receive different scores.
  \item A divergence-point detector that compares an observed trace with a higher-scoring reference trajectory. It reports four kinds of divergence: \emph{Ordering}, \emph{Manual-Refactor}, \emph{Rework}, and \emph{Hygiene}.
  \item A 45-session hand-recorded dataset on a purpose-built Java codebase with multi-label ground-truth annotations, together with a 30-session randomised-arm user study and a 48-session agent extension on a separate codebase. The recording protocol, fixtures, and analysis scripts are released alongside the report.
  \item An empirical evaluation of the score formula and the detector. This includes sensitivity and ablation tests for the score, per-kind precision and recall against inter-rater-agreed labels, and a directional comparison of how participants' refactoring process changes when they see dashboard feedback between sessions.
\end{itemize}

\section{Approach and Scope}

The approach is intended for realistic refactoring sessions, where IDE-supported refactorings and manual edits are often mixed together. We focus on Java, since it gives us mature parsing tools, established quality metrics, and refactoring-mining support such as RefactoringMiner. A refactoring session is represented as a sequence of code snapshots. Each step may correspond to a burst of manual edits or to the use of an automated IDE refactoring tool, and the same model could later be applied to traces at different levels of granularity where that data is available.

The work is not trying to search the full space of possible refactoring sequences, or to prove that the generated trajectory is globally optimal. Instead, the analysis looks for a plausible higher-scoring reference path between the same start and end states, and uses that path to explain the developer's observed process.

The work is scoped mainly to human developer sessions. The agent comparison in Chapter~\ref{ch:results} is a feasibility extension using the same tooling, not a primary evaluation target. This matters because the detectors are built around human edit patterns, such as IDE refactoring invocations, edit bursts, and commit-cadence windows, which do not map cleanly onto agent traces. Chapter~\ref{ch:results} treats that mismatch as part of what the extension reveals.

\section{Thesis Roadmap}
The rest of the thesis is organised as follows. Chapter~\ref{ch:background} reviews related work on refactoring evaluation, mining, and recommendation, and positions this project against the closest existing tools. Chapter~\ref{ch:methodology} defines the process-quality metric, the divergence-point detector, the four per-kind synthesisers, and the IntelliJ plugin and dashboard that capture and display the data. Chapter~\ref{ch:results} presents the evaluation: the sensitivity and ablation tests on the score formula, the per-kind precision and recall against externally labelled divergence kinds, the 30-session user study, and the 48-session agent extension. Chapter~\ref{ch:conclusion} draws the headline findings together, sets out the limitations they should be read against, and names the directions that would most strengthen the main results.
